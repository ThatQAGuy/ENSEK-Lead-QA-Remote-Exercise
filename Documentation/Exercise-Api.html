API Testing Exercise

Exercise Specification

Please use the swagger document https://qacandidatetest.ensek.io/ and automate the below tests around the given endpoints, using a tool or framework of your choice:

  • Reset the test data
  • Buy a quantity of each fuel
  • Verify that each order from the previous step is returned in the /orders list with the expected details
  • Confirm how many orders were created before the current date
  • Automate any other validation scenarios that you would consider writing for this API

Test Automation Solution

Generating Allure Report

Details on how to install Allure can be found here.

Once you have run the tests, navigate to .\bin\Debug\net9.0 and open a command prompt

Tip

for windows users, you can do this by typing cmd in the address bar of file expolorer

Enter allure serve, this will start Allures webserver and open a new browser tab to display the report.

Assumptions

/Login

Endpoint does not require a user to be authenticated
a succesfull login request returns a 200 response code with a message in the returned body of success and an access token
An access token is 296 characters long
An access token is valid for 24 hours

/Reset

Endpoint requires a user to be authenticated

/Orders

Endpoint requires a user to be authenticated
Updating an order quantity will not effect the current stock quantity

/Energy

Endpoint requires a user to be authenticated

/Buy

Endpoint requires a user to be authenticated
A user can not purchase more fuel than is in stock
A user can not purchase <= 0 units of fuel
An invalid order purchase will return a Bad Request response
Monetary values shall be displayed to 2 decimal places

Additional Thoughts

Improvements To Be Made

if I had more time
  • Allure Report - Create some helper fuctions to reduce the repeat code throughout the tests and improve readability
  • JSON Helpers - Create some helper fuctions to reduce the repeat code throughout the tests and improve readability
  • RestResponse - Create a generic response model so that tests doesn't directly reference RestResponse to ensure complete decoupling of RestSharp from the test to allow the HTTPClient to be easily replaced in the future shoud the need arise.
  • Githib Actions - Extend the current github action to also run the tests on a push to master
  • Githib Actions - Extend the current github action to also generate a new Allure report based on the most recent run
  • Database Access - Introduce direct calls to the database for validation to reduce flakyness of tests relying on the item under test confirming an expected result is valid
  • Enviroment Configuration - Introduce a configuration manager to handle static resources that change depending on the enviroment to allow the test to be run in multiple environments
  • Automatic defect logging - when a defect is detected automatically create or update an existing defect report
